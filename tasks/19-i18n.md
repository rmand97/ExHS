# Task 19 — Internationalization

## Goal
Danish as primary language, English as secondary. All user-facing text routed through gettext. Per-forening locale (foreninger may choose to operate in English).

## Prerequisites
- Task 14 (design system, so components support translated strings cleanly)

## Plan

### Gettext setup
- [ ] Confirm `ExhsWeb.Gettext` module wired (already present)
- [ ] Default locale: `:da`
- [ ] Supported locales: `:da`, `:en`
- [ ] Plug to set locale per-request based on user preference / forening default / Accept-Language

### Translation conventions
- [ ] All UI strings via `gettext("...")`, `ngettext(...)`, `dgettext(domain, "...")`
- [ ] No string concatenation in translations; use interpolation `%{var}`
- [ ] Domain split: `default`, `errors`, `emails`, `admin`
- [ ] Add to `AGENTS.md`: never hardcode user-facing English; agents must wrap in gettext

### Translation workflow
- [ ] `mix gettext.extract` updates `.pot` files
- [ ] `mix gettext.merge priv/gettext` updates `.po` files
- [ ] CI check: no missing translations in `da` for any new pot string

### User locale preference
- [ ] Add `locale` attribute to User (atom: `:da | :en`)
- [ ] Profile UI lets user choose
- [ ] Emails sent in user's locale

### Forening default locale
- [ ] Add `default_locale` attribute to Forening
- [ ] Public pages use forening locale (overrideable via query param or visitor preference)

### Content translation (separate from UI strings)
- [ ] Event titles/descriptions, newsletter bodies, product descriptions — these are user-generated content; not gettext
- [ ] Optional: per-forening multilingual content (e.g. event has `title_da`, `title_en`)? **TBD** — see open decisions

### Email templates
- [ ] All transactional + marketing email templates support both locales
- [ ] Subject lines too

### Number / date / currency formatting
- [ ] Use `Cldr` (`ex_cldr`) for locale-aware formatting
- [ ] Danish: comma decimal separator, period thousands separator
- [ ] DKK formatted as `123,45 kr.`

### Tests
- [ ] Setting locale via plug works
- [ ] Email renders in correct locale
- [ ] Currency/date formatting matches locale

## Open decisions
- [ ] **User-generated multilingual content** — bilingual event titles needed?
- [ ] **More locales later** — German? Swedish? Plan for extensibility but don't build now
- [ ] **CLDR vs hand-rolled formatters** — CLDR is heavy but correct

## Done when
- App fully usable in both Danish and English
- All emails respect user locale
- `mix gettext.extract && mix gettext.merge priv/gettext` produces no diff in CI
